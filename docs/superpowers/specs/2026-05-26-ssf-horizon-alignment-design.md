# SSF Alphapurify 持有期窗口对齐设计

## 背景

当前仓库中的 3 个社保基金 Alphapurify 分析脚本已经可以从数据库构造因子面板并运行分析：

1. `factor/alphapurify_ssf_count.py`
2. `factor/alphapurify_ssf_ratio.py`
3. `factor/alphapurify_ssf_ratio_change.py`

但当前实际输出结果仍然是短周期日频窗口（如 `1D / 5D / 10D`），与用户明确要求的评估口径 **1个月 / 1季度 / 半年 / 1年** 不一致。这个偏差已经影响到评估文档，导致“用户要求、脚本输出、文档表述”三者没有保持一致。

## 目标

把 3 个 SSF Alphapurify 分析脚本的未来收益评估窗口统一调整为：

1. `1个月 = 20` 个交易日
2. `1季度 = 60` 个交易日
3. `半年 = 120` 个交易日
4. `1年 = 240` 个交易日

并确保以下内容保持一致：

1. 脚本实际运行口径
2. 测试断言
3. 评估文档中的结果展示和描述

## 范围

本次改动包含：

1. 为 SSF Alphapurify 分析引入统一的默认持有期窗口定义。
2. 让 3 个 SSF 脚本都显式使用 `20 / 60 / 120 / 240`。
3. 更新脚本测试，验证传给 `FactorAnalyzer` 的窗口口径。
4. 重跑 3 个因子分析，并更新 3 份评估文档。

本次改动不包含：

- 改动 `ssf_count` / `ssf_total_hold_ratio` / `ssf_total_hold_ratio_change` 的因子定义。
- 改动社保基金识别规则。
- 改动公告日到交易日的映射逻辑。
- 为所有其他 Alphapurify 因子统一改窗口；本次仅修正 SSF 相关脚本。

## 方案对比

### 方案 A：直接在 3 个脚本里写死 20/60/120/240

优点：

- 改动最小，能最快对齐这次需求。

缺点：

- 3 个脚本会重复保存同一组窗口常量。
- 以后如果再次调整口径，容易漏改。

### 方案 B：抽出共享默认窗口常量，由 3 个脚本统一复用（推荐）

优点：

- 保持 3 个脚本口径完全一致。
- 后续如需调整 SSF 分析窗口，只需要改一处。
- 测试可以围绕统一常量断言，减少重复硬编码。

缺点：

- 比方案 A 多一层抽象，但复杂度很低。

### 方案 C：新增 CLI 参数允许自定义窗口，同时默认 20/60/120/240

优点：

- 灵活性最高。

缺点：

- 需求当前只要求固定口径，多出来的配置会增加测试和文档复杂度。
- 这次主要问题是默认口径不一致，而不是缺少可配置性。

## 推荐方案

采用 **方案 B**：

1. 在 SSF Alphapurify 分析层定义统一默认窗口，例如 `DEFAULT_FORWARD_RETURNS = [20, 60, 120, 240]`。
2. 3 个脚本在构造 `FactorAnalyzer` 时统一传入这组窗口。
3. 脚本、测试、文档全部围绕同一组窗口同步更新。

## 详细设计

### 1. 共享窗口定义

在 SSF 分析相关模块中新增一组共享默认窗口常量，语义明确对应：

- `20` → `1个月`
- `60` → `1季度`
- `120` → `半年`
- `240` → `1年`

该常量只负责“未来收益评估窗口”，不参与因子计算本身。

### 2. 脚本层改动

3 个脚本都保持当前 CLI 和数据准备流程不变，只调整 `run_analysis()` 中构造 `FactorAnalyzer` 的参数，使其显式使用上述窗口。

保持不变的部分：

1. 通过 DB 读取行情和 `top10_floatholders`。
2. 通过 `build_ssf_factor_panel_from_db()` 生成面板。
3. 校验 `datetime / symbol / close / factor` 必要列。
4. 输出面板规模、mean IC、HTML 报告路径。

调整后的行为是：脚本实际计算的未来收益 horizon 不再依赖当前短周期默认值，而固定为 `20 / 60 / 120 / 240`。

### 3. 文档口径

3 份评估文档都改为使用以下区间展示结果：

1. `1个月`
2. `1季度`
3. `半年`
4. `1年`

文档中的结果值必须来自新的实际运行结果，不能沿用旧的 `1D / 5D / 10D` 输出换标签。

### 4. 测试策略

脚本测试在现有基础上补充一层验证：

1. 伪造 `FactorAnalyzer`。
2. 捕获构造参数。
3. 断言 3 个脚本都把相同的窗口值传入分析器。

共享层测试不需要因为这次改动而扩大范围，因为因子计算和面板构造逻辑本次不变。

### 5. 错误处理

继续保持现有显式失败策略：

1. `alphapurify` 不可用时抛出安装提示。
2. 面板为空时返回错误退出。
3. 缺少 `datetime / symbol / close / factor` 关键列时抛出 `ValueError`。

本次改动不引入静默 fallback，也不改变现有报错路径。

## 验证要求

实现完成后至少需要验证：

1. `test/factor/test_alphapurify_ssf_count.py`
2. `test/factor/test_alphapurify_ssf_ratio.py`
3. `test/factor/test_alphapurify_ssf_ratio_change.py`
4. 相关脚本在真实数据上重新运行，得到新的 20/60/120/240 结果
5. 3 份文档与脚本实际输出保持一致

## 预期结果

修正完成后，用户看到的 SSF 因子分析输出将统一为：

- `1个月`
- `1季度`
- `半年`
- `1年`

不再出现脚本按 `1D / 5D / 10D` 运行、文档却按月度口径描述的错位问题。
