# 小黑屋全局禁买设计

## 背景

当前仓库里的选股、调仓、监控和交易逻辑大多分散在不同策略、脚本和服务中。用户希望引入一个**全局约束**：

1. 某只股票一旦进入“小黑屋”，在禁买期内所有策略都不得再次买入
2. 选股结果需要自动剔除小黑屋股票
3. 已持仓股票不强制卖出，只限制后续新买入
4. 第一版通过 CLI 手动维护
5. 未来希望通过扫描“大股东减持信息”自动加入

这个能力如果散落在各个策略里单独实现，会很快变成“有的策略接了、有的没接”的局面，无法满足 global 约束。

## 目标

本次设计目标是增加一个统一的小黑屋能力，使其成为所有买入流程都必须遵守的全局规则：

1. 提供持久化的小黑屋记录，而不是临时内存名单
2. 每条记录可单独配置禁买天数，并自动到期失效
3. 提供统一服务判断单只股票是否禁买，并批量过滤候选列表
4. 提供 CLI 做增删改查和状态查看
5. 为未来“自动扫描后加入小黑屋”预留复用入口

## 范围

本次改动包含：

1. 新增小黑屋数据模型与存储接口
2. 新增小黑屋服务层，封装校验、到期判定和批量过滤
3. 新增 CLI 管理入口
4. 在策略/选股买入路径中接入统一过滤器，使小黑屋规则全局生效
5. 增加对应的单元测试和集成测试

本次改动不包含：

- 自动扫描大股东减持并写入小黑屋
- 根据进入小黑屋事件触发强制卖出
- 改动现有策略的核心选股逻辑或调仓节奏
- 改动 DAG 调度、任务边界或运行频率

## 方案对比

### 方案 A：各策略各自查询并过滤

特点：

- 每个策略在自己的选股逻辑里查询小黑屋
- 直接在本地剔除禁买股票

优点：

- 看起来接入很直接

缺点：

- 很容易遗漏某些策略或脚本
- 同样的过滤逻辑会复制到多个地方
- 后续自动加入或规则变更时维护成本高

### 方案 B：配置文件静态黑名单

特点：

- 通过配置文件维护禁买股票列表
- 策略读取配置后自行过滤

优点：

- 初期实现成本低

缺点：

- 不适合“每条记录独立 ban_days、自动到期”的需求
- 历史记录和来源追踪能力弱
- 不利于未来接自动扫描入口

### 方案 C：统一小黑屋数据源 + 公共过滤接口 + CLI（推荐）

特点：

- 小黑屋记录统一落库
- 服务层提供单股判定和批量过滤
- CLI 负责人工维护
- 策略统一复用公共过滤器

优点：

- 真正满足 global 约束
- 手动维护和未来自动加入可共用同一入口
- 规则集中，测试和后续扩展都更清晰

缺点：

- 第一版需要同时改存储、服务、CLI 和若干买入入口

## 详细设计

### 1. 数据模型

新增一张小黑屋表，按“单条禁买事件”建模，而不是只保存一个当前名单。

建议字段：

1. `id`
2. `stock_code`
3. `market`
4. `ban_days`
5. `start_at`
6. `expire_at`
7. `source`
8. `note`
9. `enabled`
10. `created_at`
11. `updated_at`

字段语义：

- `stock_code` / `market`：标识证券
- `ban_days`：该条记录的禁买时长，单位天
- `start_at`：禁买开始时间，默认创建时间
- `expire_at`：到期时间，新增或更新时由 `start_at + ban_days` 计算
- `source`：记录来源，第一版使用 `manual`，后续可扩展成 `shareholder_reduction`
- `enabled`：显式开关；即使未到期，也可以手动禁用
- `note`：记录原因或备注

同一股票允许存在多条历史记录。判定是否禁买时，只要存在任一条 `enabled=true` 且当前时间早于 `expire_at` 的记录，即视为仍在小黑屋。

### 2. 存储层

在 `storage/model/` 中新增对应模型，并在 `storage/storage_db.py` 中补充以下能力：

1. 建表或确保表存在
2. 创建小黑屋记录
3. 更新小黑屋记录
4. 删除小黑屋记录
5. 查询单条记录
6. 列表查询
7. 查询某只股票当前是否有有效禁买记录
8. 批量查询一组股票中哪些处于禁买状态

存储层负责数据读写和基础查询，不承载复杂业务拼装。时间窗口判定可以由 SQL 直接筛选“当前有效记录”，也可以先查询后由服务层统一封装，但公开给上层的能力应明确表达“当前是否有效”。

### 3. 服务层

新增独立的小黑屋服务，例如 `monitor/` 下与现有管理服务同风格的服务类，或放在更适合全局买入规则的位置。服务层职责如下：

1. 参数校验
2. 统一创建/更新 payload
3. 计算 `expire_at`
4. 提供 `is_buy_banned(stock_code, market, now=None)` 能力
5. 提供 `filter_buy_candidates(stock_codes, market, now=None)` 能力
6. 产出稳定的返回结构，便于 CLI 和后续自动化入口复用

行为规则：

1. 新增记录时默认 `source="manual"`
2. `ban_days` 必须为正整数
3. `start_at` 默认当前时间
4. `expire_at` 根据 `ban_days` 自动计算
5. 若记录已过期，则视为失效，不需要手工删除
6. 若过滤器执行失败，不应静默放行，应显式暴露错误

### 4. 全局过滤接入

核心原则：**策略负责产生候选，小黑屋负责判定买入资格。**

因此不建议每个策略手写黑名单判断，而是统一调用公共过滤接口。接入方式如下：

1. 在所有会生成“最终待买入列表”的路径里调用 `filter_buy_candidates(...)`
2. 若某些路径是直接对单只股票下单，则在下单前调用 `is_buy_banned(...)`
3. 已持仓股票不因进入小黑屋而被强制卖出
4. 选股结果里命中的股票需要被自动剔除

优先寻找现有的公共买入/选股入口收敛接入；如果当前仓库没有完整统一入口，则先把当前明确存在的策略和工具路径接到一个共享 helper 上，避免逻辑复制。

### 5. CLI 设计

第一版通过 CLI 手动维护，因此命令风格应尽量复用当前 `stock_monitor` 这类命令的约定，包括稳定 JSON 输出、人类可读输出和明确退出码。

建议命令：

1. `blackroom add --stock-code 600519 --market A --ban-days 30 --note "..." [--start-at ...]`
2. `blackroom update --id 1 [--ban-days 45] [--note "..."] [--enabled|--disabled]`
3. `blackroom remove --id 1`
4. `blackroom list [--active-only] [--stock-code ...]`
5. `blackroom get --id 1`
6. `blackroom status`

如果仓库已有统一股票管理 CLI，更合适的做法是把小黑屋作为其子命令，而不是新增完全独立的命令风格。

### 6. 未来自动加入扩展

未来当“大股东减持扫描”落地时，不直接写数据库表，而是复用服务层的新增入口：

1. 扫描模块识别到符合条件的减持事件
2. 组装 `stock_code / market / ban_days / source=shareholder_reduction / note`
3. 调用统一服务创建记录
4. 其余策略仍通过同一个过滤器生效

这样可以避免“人工添加”和“自动添加”走两套不同逻辑。

## 错误处理

本次设计坚持显式失败，不做静默 fallback：

1. CLI 参数错误返回 `VALIDATION_ERROR`
2. 记录不存在返回 `NOT_FOUND`
3. 存储层异常继续显式报错或返回一致的失败结构
4. 过滤器调用失败时，策略侧不默认放行

这样虽然更严格，但能保证 global 规则不会因为一次查询失败而被绕过。

## 测试策略

至少覆盖以下验证：

1. 数据模型和存储层：新增、更新、删除、查询、到期判定
2. 服务层：`ban_days` 校验、`expire_at` 计算、单股禁买判断、批量过滤
3. CLI：增删改查、状态查看、JSON 输出、错误码
4. 策略接入：小黑屋股票会从待买入列表中剔除；已持仓股票不会被强制卖出
5. 未来扩展点：保留 `source` 字段和统一服务入口，确保后续自动扫描可以无缝接入

## 验收标准

满足以下条件即可认为设计落地成功：

1. 小黑屋规则在存储层有独立持久化模型
2. 每条记录支持独立 `ban_days`，并自动到期失效
3. CLI 可以完成手动增删改查
4. 至少现有主要买入/选股路径统一复用公共过滤接口
5. 小黑屋股票会被自动剔除，但不会触发强制卖出
6. 后续自动扫描模块可以复用同一新增入口，而不需要重写规则
