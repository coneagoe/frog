# 社保基金前十大流通股东变动跟踪提醒设计

## 背景

仓库当前已经具备两块可复用能力：

1. `top10_floatholders` 周度下载与入库链路已存在。
2. 邮件发送能力与 DAG / task 组织方式已存在。

但现有 `monitor` 主链路面向的是价格、涨跌幅、均线、RSI 等条件触发；本次需求是基于公告期之间的 `top10_floatholders` 数据，识别社保基金在前十大流通股东中的持仓变化，并生成排序后的一次性汇总提醒。

这两类问题的触发方式、状态管理和输出形态不同，不适合把“社保持仓变动”硬塞进当前 `monitor_runner.py`。

## 目标

在**不改现有 `monitor` 主链路**的前提下，新增一条独立的周度分析链路，实现以下闭环：

1. 在 `top10_floatholders` 周更下载完成后，自动分析每只股票最新公告期相对上一公告期的社保基金持仓变化。
2. 识别 **新进 / 增持 / 减持 / 退出** 四类事件。
3. 对股票级信号按“事件强弱 + 出现家数 + 集中度”计算综合分并排序。
4. 仅当本轮出现新增信号时，发送一封按分数排序的汇总邮件。
5. 将分析结果持久化，支持幂等、回看和后续调参。

## 范围

本次改动包含：

1. 新增社保基金股东识别逻辑。
2. 新增两期 `top10_floatholders` 差分分析逻辑。
3. 新增股票级评分与排序逻辑。
4. 新增社保持仓变动信号表及对应存取方法。
5. 在现有周更 top10 DAG 末尾追加分析入口与汇总邮件发送逻辑。
6. 新增相关单元测试 / 任务层测试。

本次改动不包含：

- 修改现有 `monitor.condition`、`monitor.monitor_runner`、`monitor_target` 数据模型。
- 一股一封的即时邮件提醒。
- 图形化看板、自定义权重界面、多渠道推送。
- 复杂机器学习评分或市场因子混合模型。
- 改动 `download_top10_floatholders_weekly` 的既有 schedule、重试、并发分片策略。
- 新建第二个独立定时分析 DAG 作为默认实现。

## 设计原则

### 不改现有 monitor 主链路

本需求是“公告期差分 + 批量排序 + 汇总提醒”，不是“单标的实时条件触发”。因此：

- 不扩展现有 `ConditionResult` / `evaluate_condition()`
- 不把社保持仓变化塞入 `monitor_target.condition`
- 不复用当前 `last_state` 式边沿触发模型

只复用公共能力：入库数据、`storage` 风格、邮件发送能力、DAG / task 组织模式。

### 数据驱动优于日历空跑

当前 `top10_floatholders` 是周更数据，因此分析任务应在现有周更下载完成后触发，而不是每日固定空跑。

推荐节奏：

1. 现有周更 DAG 的分片下载任务全部完成
2. 同一 DAG 末尾立刻运行一个社保持仓变动分析 task
3. 若本轮有新增信号，则发送一封汇总邮件

### 股票级结果优于股东级碎片结果

同一只股票同一期可能对应多个社保基金主体。邮件提醒和排序应以**股票级信号**为中心，同时保留股东明细供回溯和展示。

### 幂等与显式失败

- 同一股票同一公告期只生成一条股票级信号
- 重复运行任务时不重复插入、不重复提醒历史结果
- 邮件发送失败要显式失败，不能静默吞掉

## 详细设计

### 1. 社保基金识别层

新增一个独立识别模块，对 `top10_floatholders.holder_name` 做规则匹配，筛出社保基金股东。

第一版采用名称规则识别即可，例如匹配“全国社保基金”“社保基金”“基本养老保险基金”一类稳定关键词，并保留扩展点，便于后续补充白名单 / 正则规则。

输出为标准化的社保基金股东记录集合，不在该层做评分或聚合。

### 2. 变动分析层

以 `stock_id` 为单位，读取该股最新公告期和上一公告期的 `top10_floatholders` 数据，只对社保基金股东子集进行比较。

事件定义如下：

1. **新进**：上一期不存在，本期进入前十大流通股东
2. **增持**：两期均存在，且持股数量上升，或 `hold_change > 0`
3. **减持**：两期均存在，且持股数量下降，或 `hold_change < 0`
4. **退出**：上一期存在，本期不存在

特殊情况：

- 若某股票只有一期数据，则跳过，不生成信号
- 若上一期和本期都没有识别到社保基金，则不生成信号
- 若上一期有社保基金、本期没有，则生成“退出”型信号

### 3. 股票级聚合层

股东级事件生成后，按股票聚合为单条股票级信号。建议至少生成以下指标：

- `ssf_holder_count_now`
- `ssf_holder_count_prev`
- `ssf_holder_count_change`
- `ssf_total_hold_ratio_now`
- `ssf_total_hold_ratio_prev`
- `ssf_total_hold_ratio_change`
- `max_single_holder_ratio_now`
- `event_types`
- `detail_json`

其中：

- `ssf_holder_count_*` 表示社保基金主体家数
- `ssf_total_hold_ratio_*` 表示社保基金合计持股比例
- `detail_json` 保存股东级明细，用于邮件展示和回溯

### 4. 评分与排序层

第一版采用可解释、易调参的线性评分，不叠加市场价格、成交额等外部因子。

建议总分为 100 分制，包含三部分：

1. **事件分（50%）**
   - 新进：`+1.0`
   - 增持：`+0.7`
   - 减持：`-0.7`
   - 退出：`-1.0`
2. **家数分（20%）**
   - 社保基金家数越多、相对上一期增加越多，得分越高
3. **集中度分（30%）**
   - 社保基金合计持股比例越高、相对上一期提升越多，得分越高

可按如下思路落地：

```text
score = 50 * event_score + 20 * count_score + 30 * concentration_score
```

要求：

- 同时保留总分和分项分，便于邮件解释和后续调权重
- 排序按 `score DESC`
- 第一版优先保证稳定、可解释，不追求金融模型最优

### 5. 信号落库层

新增一张独立的社保持仓变动信号表，存放分析结果历史。该表不复用 `monitor_target`。

建议字段至少包括：

- `stock_id`
- `ann_date`
- `prev_ann_date`
- `event_types`
- `score`
- `ssf_holder_count_now`
- `ssf_holder_count_prev`
- `ssf_holder_count_change`
- `ssf_total_hold_ratio_now`
- `ssf_total_hold_ratio_prev`
- `ssf_total_hold_ratio_change`
- `detail_json`
- `created_at`

唯一约束建议使用：

- `stock_id + ann_date`

含义是“同一股票同一最新公告期只允许存在一条股票级信号”。这样任务重复运行时可以幂等跳过或 upsert，不会重复提醒历史结果。

### 6. 提醒层

提醒策略采用“**每轮分析最多一封汇总邮件**”：

1. 现有 `download_top10_floatholders_weekly` DAG 的全部下载分片成功后运行 analyzer
2. analyzer 只处理尚未生成过信号的最新公告期股票
3. 若本轮新增信号数为 0，则不发邮件
4. 若本轮新增信号数大于 0，则发送一封汇总邮件

邮件主题建议：

```text
[社保持仓变动提醒] 2026-05-18 新增 12 只，按综合分排序
```

邮件正文至少包含：

- 股票代码
- 事件类型
- 综合分
- 社保基金家数变化
- 社保基金合计持股比例变化
- 主要社保基金主体明细（可截断）

正文先展示 Top N，再附完整榜单或摘要。

### 7. 代码落点

建议将新功能做成独立模块，不接入现有 `monitor_runner.py`。

一种符合当前仓库风格的拆分方式如下：

- `shareholder_monitor/ssf_detector.py`
  - 负责社保基金股东识别
- `shareholder_monitor/ssf_change_analyzer.py`
  - 负责两期差分、聚合、评分
- `shareholder_monitor/ssf_alert.py`
  - 负责邮件正文拼装
- `storage/model/ssf_change_signal.py`
  - 信号表 ORM
- `storage/storage_db.py`
  - 信号表建表、保存、查询方法
- `task/` 或 `tools/`
  - 暴露分析入口，供现有 DAG 或命令行复用
- `dags/download_top10_floatholders_weekly.py`
  - 在现有分片下载任务之后追加一个单一 analysis task
  - 该 task 依赖全部分片任务成功完成后再执行

目录名可以在实现时根据仓库现有结构微调，但原则是不侵入当前 `monitor` 的 price-based runner，也不新增第二个定时分析 DAG 作为默认路径。

## 错误处理

沿用仓库当前“显式失败优于静默成功”的风格：

- 某股票缺少上一期数据：跳过并记录日志
- 某股票分析异常：记录错误并继续分析其他股票
- 批处理完成后若存在错误，可在任务汇总中体现失败明细
- 邮件发送失败：任务显式失败抛错
- 表写入失败：显式返回失败 / 抛错，不静默忽略

## 测试策略

### 单元测试

1. **识别层**
   - 正确识别社保基金名称
   - 非社保基金股东不误判

2. **差分层**
   - 新进 / 增持 / 减持 / 退出 四类场景
   - 同股多个社保基金主体的聚合
   - 无上一期数据时跳过

3. **评分层**
   - 事件权重生效
   - 家数变化影响得分
   - 集中度变化影响得分

4. **存储层**
   - 同一 `stock_id + ann_date` 幂等写入
   - `detail_json` 等字段正确保存

### 任务层测试

1. 有新增信号时发送汇总邮件
2. 无新增信号时不发送邮件
3. 单只股票失败不阻断整个批次分析
4. 重复运行不会重复发送历史提醒

## 验收标准

满足以下条件即视为完成：

1. 现有周更 top10 DAG 分片任务完成后，能自动运行一个下游社保持仓变动分析 task。
2. 能识别新进、增持、减持、退出四类社保基金事件。
3. 能按股票输出综合分与排序结果。
4. 同一股票同一公告期不会重复生成信号。
5. 有新增信号时会发送一封按分数排序的汇总邮件；无新增信号时不发。
6. 不修改现有 `monitor_runner.py` 的行为。

## 备选方案评估

### 方案一：复用现有 monitor condition / runner

优点：表面上复用更多现有代码。  
缺点：现有 monitor 面向价格型条件，不适合公告期批量差分、排序和汇总邮件；会把边界搞混。  
结论：不采用。

### 方案二：新增独立周度社保持仓变动分析链路

优点：与需求匹配最好，边界清晰，便于幂等、评分、回看和后续调参。  
缺点：需要新增少量领域模块和结果表。  
结论：采用。

### 方案三：只做离线榜单导出，不做提醒

优点：实现最快。  
缺点：缺少提醒闭环，不满足“跟踪提醒”目标。  
结论：不采用。
