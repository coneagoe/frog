# 社保持仓统一周报设计

## 背景

当前 `download_top10_floatholders_weekly` DAG 在完成前十大流通股东下载后，会运行 `shareholder_monitor.runner.run_ssf_change_alert()`。

现有实现虽然是**周度触发**，但发信策略是**按公告日期分组逐封发送**：

1. 查询所有待发送 `ssf_change_signals`
2. 按 `ann_date` 分组
3. 每个公告日期生成一封邮件
4. 每封邮件内再按单条信号 `score` 排序

这与“给我发一份统一的周报，打分也合在一起打”的目标不一致。

## 目标

将当前“按公告日期拆分多封邮件”的行为改为：

1. 每次周度任务运行，最多发送 **1 封统一周报**
2. 周报内包含本次所有待发送的社保持仓变动信号
3. 不引入新的周报级评分模型
4. 继续复用每条信号已有的 `score`
5. 对整周全部待发送信号做 **全局统一排序**

## 范围

本次改动包含：

1. 调整 `run_ssf_change_alert()` 的待发送聚合方式
2. 调整 `build_ssf_change_alert_email()` 的主题与正文格式
3. 调整 runner / alert 层测试，覆盖“只发一封统一周报”

本次改动不包含：

- 修改 `top10_floatholders` 下载逻辑
- 修改单条信号分析和 `score` 计算公式
- 引入新的周报综合评分、行业聚合、市场因子
- 修改 DAG schedule、retry、partition 结构
- 新增日报、月报或按股票单独提醒

## 方案对比

### 方案 A：统一周报，全部混排后按现有 score 全局排序（推荐）

特点：

- 只发一封邮件
- 不再按 `ann_date` 分段
- 正文中每条记录保留 `ann_date`
- 使用已有 `score` 直接做全量降序排序

优点：

- 改动最小
- 不改变分析语义
- 最符合“打分也合在一起打”的确认结果

缺点：

- 不同公告日期的记录会混排，日期感知主要依赖正文列出

### 方案 B：统一周报，但正文内按公告日期分段

特点：

- 仍只发一封邮件
- 邮件正文保留多个 `ann_date` 分段
- 各分段内排序

优点：

- 更容易看出每条信号对应哪个公告日期

缺点：

- 不符合“全部混排后只按总分排序”的确认结果

### 方案 C：统一周报并重新定义周报级评分

特点：

- 对不同公告日期或不同事件额外加权
- 再按新的周报级分数排序

优点：

- 可扩展性高

缺点：

- 会改变现有信号解释方式
- 超出本次需求，且需要额外设计和回归验证

## 详细设计

### 1. runner 聚合逻辑

当前 `run_ssf_change_alert()` 在取到 `pending = storage.list_pending_ssf_change_signals()` 后，会按 `ann_date` 分组并循环发送。

改动后：

1. 保留候选分析、信号入库逻辑不变
2. 保留 `pending` 查询逻辑不变
3. 删除按 `ann_date` 分组发送的分支
4. 直接把完整 `pending` 列表交给邮件构建函数
5. 一次调用 `send_email(subject, body)`
6. 一次调用 `storage.mark_ssf_change_signals_alerted([...all ids...])`

这样每次任务运行只会产生一封周报。

### 2. 邮件构建逻辑

`build_ssf_change_alert_email()` 改为接收整周待发送信号列表，并在内部按单条 `score` 降序排序。

正文中每条记录至少展示：

- `stock_id`
- `ann_date`
- `score`
- `event_types`
- `ssf_holder_count_change`
- `ssf_total_hold_ratio_change`

主题建议改为统一周报语义，例如：

```text
[社保持仓周报] 本周新增 12 只，按综合分排序
```

正文头部建议增加摘要，例如：

```text
待发送信号数: 12
公告日期范围: 2024-03-31 ~ 2024-06-30
排序规则: 按单条信号综合分降序
```

### 3. 打分语义

本次不改 `shareholder_monitor.ssf_change_analyzer.analyze_ssf_change()`。

明确约束如下：

1. 单条信号 `score` 的计算方式完全保持不变
2. “打分也合在一起打”的含义是**全量待发送信号使用同一排序池**
3. 不新增周报级 score、日期权重、衰减因子或二次排序模型

换句话说，统一的是**排行榜**，不是**评分公式**。

### 4. 状态与失败语义

保留现有可靠性语义：

1. 如果没有 pending 信号，则不发邮件
2. 如果 `send_email()` 失败，则本批信号保持 pending，等待下次重试
3. 只有在邮件发送成功后，才统一标记 `alerted`
4. 如果标记已发送失败，则下次重试可能再次发送同一封周报；该行为与当前实现保持一致

### 5. 代码落点

本次改动预计集中在：

- `shareholder_monitor/runner.py`
  - 从“按公告日期循环发送”改为“单次统一发送”
- `shareholder_monitor/ssf_alert.py`
  - 主题调整为周报语义
  - 正文补充 `ann_date` 字段和整体摘要
- `test/shareholder_monitor/test_runner.py`
  - 将“按公告日期发 2 封”的测试改为“只发 1 封并包含多个日期”
- `test/shareholder_monitor/test_ssf_alert.py`（若已有或需要补充）
  - 锁定统一周报格式与全局排序行为

## 错误处理

本次改动不改变错误处理原则：

- 分析阶段单股异常：继续统计其他股票，并计入失败数
- 无 pending：正常返回，不发信
- 发信失败：显式抛错，不标记已发送
- 标记发送失败：显式抛错，允许后续重试导致重复发送

## 测试策略

至少覆盖以下场景：

1. 多个 `ann_date` 的 pending 信号只发送 **1 封**
2. 邮件正文同时包含多个公告日期记录
3. 邮件正文按全量 `score` 降序排序，而非按日期分段
4. 发信成功后，一次性标记全部已发送 ID
5. 发信失败时，不标记任何记录

## 验收标准

满足以下条件即可认为完成：

1. 周度 DAG 分析 task 每次运行最多发送 1 封邮件
2. 邮件中包含所有 pending 社保持仓信号
3. 排序基于整批信号现有 `score` 全局降序
4. 邮件中每条记录都能看出对应的 `ann_date`
5. 现有“发送失败不标记”的语义未被破坏
