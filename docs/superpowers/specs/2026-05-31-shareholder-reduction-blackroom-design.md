# 股东减持公告入小黑屋设计

## 背景

当前仓库已经具备：

1. 持久化的小黑屋记录表 `blackroom_records`
2. 统一的小黑屋服务 `BlackroomManagementService`
3. `stock-monitor blackroom ...` CLI 管理入口
4. `source=shareholder_reduction` 的来源枚举预留

现阶段，小黑屋主要依赖手工录入。用户希望新增一个能力：**根据股东减持公告，把对应股票自动放进小黑屋**，但第一版不直接接入定时任务，而是先提供一个**可手动执行的同步命令/脚本**。

## 目标

本次设计目标：

1. 基于 Tushare 的股东减持公告数据，识别命中的股票
2. 将命中的股票写入现有小黑屋体系，而不是新建并行黑名单逻辑
3. 默认禁买时长为半年
4. 若股票当前已在有效小黑屋中，则跳过，不重复新增
5. 提供可重复执行、可审计的手动同步入口

## 非目标

本次不包含：

- 接入 DAG、定时任务或后台常驻扫描
- 根据公告内容做 NLP、PDF 解析或标题关键词识别
- 强制卖出已持仓股票
- 自定义“重要股东/大股东”规则的二次判定

## 需求确认

根据本次确认，第一版采用以下口径：

1. 触发源：**股东减持公告**
2. 主体判定：**直接按 Tushare 接口返回口径**
3. 封禁时长：**半年**
4. 运行方式：**先做手动同步命令**
5. 重复处理：**若股票已在有效黑屋中则跳过**

## 方案对比

### 方案 A：基于 `stk_holder_reduce` 做手动同步命令（推荐）

特点：

- 使用 Tushare `stk_holder_reduce`
- 按公告日期区间拉取减持数据
- 提取股票代码并写入现有小黑屋

优点：

- 与“股东减持公告”目标最贴合
- 不需要自行从全量公告标题中做关键词筛选
- 可以直接复用现有 blackroom service 和存储模型

缺点：

- 依赖该接口权限与积分
- 数据质量完全依赖 Tushare 接口定义

### 方案 B：基于 `stk_holdertrade` 并按减持方向过滤

特点：

- 使用更通用的增减持接口
- 通过减持方向字段筛选出减持记录

优点：

- 可覆盖更广的股东变动场景

缺点：

- 口径更偏“持股变动”而非“减持公告”
- 字段和语义更容易混入非目标场景

### 方案 C：基于 `anns_d` 抓公告再做关键词筛选

特点：

- 拉取全量公告
- 根据标题或后续文本识别“减持公告”

优点：

- 最灵活，未来可扩展更多事件类型

缺点：

- 噪音大
- 需要维护关键词规则
- 第一版实现复杂度明显更高

## 推荐方案

采用**方案 A**：新增一个基于 `stk_holder_reduce` 的手动同步入口，把命中的股票写入现有小黑屋，并标记 `source=shareholder_reduction`。

这样既满足“按减持公告入小黑屋”的目标，又能保持第一版实现简单、可追踪、可复用。

## 详细设计

### 1. 同步入口

在现有 `stock-monitor` CLI 下新增一个同步子命令，而不是额外新建一套命令风格。

建议命令形态：

```bash
stock-monitor blackroom sync-shareholder-reduction \
  --start-date 20260501 \
  --end-date 20260531 \
  [--ban-days 180]
```

其中：

1. `--start-date` / `--end-date` 为必填，格式 `YYYYMMDD`
2. `--ban-days` 可选，默认 `180`
3. 第一版不额外暴露复杂过滤项，先保证能力稳定

如果后续需要接定时任务或批量历史补数，直接复用这套同步逻辑。

### 2. 数据源与拉取逻辑

同步命令内部调用 Tushare `stk_holder_reduce(start_date=..., end_date=...)`。

处理规则：

1. 只按 Tushare 返回结果判断，不再人为叠加“大股东/重要股东”自定义过滤
2. 从返回结果中提取 `ts_code`
3. 将 `ts_code` 归一成现有小黑屋使用的 `stock_code + market`
4. 同一批次内，按股票维度去重

这样同一股票在同一窗口内即使出现多条减持公告，也只会形成一次待写入动作。

### 3. 写入小黑屋

同步逻辑不直接写数据库，而是统一复用 `BlackroomManagementService.add_record(...)`。

建议写入 payload：

1. `stock_code`：由 `ts_code` 拆出代码部分
2. `market`：由 `ts_code` 后缀映射，A 股场景下通常为 `A`
3. `ban_days`：默认 `180`，允许命令行覆盖
4. `source`：固定为 `shareholder_reduction`
5. `note`：记录公告摘要，例如 `股东减持公告 20260531 / 张三`

写入前先调用现有禁买判定能力；若股票已在有效小黑屋中，则跳过，不重复新增记录。

### 4. 幂等与重复执行

第一版要求同步命令具备稳定的重复执行特性：

1. 同一时间区间重复跑，不应为同一只已在有效黑屋中的股票不断新增记录
2. 同一批结果内多条公告命中同一只股票时，只保留一次待写入动作
3. 已过期的旧黑屋记录不算“已存在”，允许重新加入

这能保证命令既能日常重复执行，也适合历史回补。

### 5. 输出与可观测性

同步命令输出应延续 `stock-monitor` 的稳定风格，至少包含：

1. `fetched`：从 Tushare 拉回的原始公告条数
2. `unique_stocks`：按股票去重后的数量
3. `added`：本次新增入黑屋的股票数
4. `skipped`：因已在有效黑屋中而跳过的股票数

如使用 `--json`，建议返回稳定 JSON 结构，便于后续脚本或自动化任务复用。

### 6. 错误处理

遵循现有仓库“显式失败”的原则：

1. Tushare token 缺失时，显式报错
2. 接口权限/积分不足时，显式报错
3. 日期格式非法或 `start_date > end_date` 时，返回参数错误
4. Tushare 返回空数据时，视为成功同步但 `added=0`
5. 存储或服务层失败时，不静默吞掉异常

不得把“权限不足”或“请求失败”伪装成“当天没有减持公告”。

### 7. 测试策略

至少覆盖以下测试：

1. 同步逻辑单测：mock Tushare 返回，验证日期参数、批内去重、跳过已在黑屋股票、正确写入 payload
2. CLI 测试：验证 `sync-shareholder-reduction` 参数解析、JSON 输出和失败退出码
3. 服务复用测试：确认自动写入仍沿用现有 `shareholder_reduction` source 和 blackroom 校验行为

测试重点不是 Tushare 本身是否正确，而是**本仓库如何消费这批数据并稳定写入小黑屋**。

## 验收标准

满足以下条件即可认为本设计落地成功：

1. 存在一个可手动执行的“减持公告同步入黑屋”入口
2. 同步数据源使用 `stk_holder_reduce`
3. 命中的股票会写入现有小黑屋，来源为 `shareholder_reduction`
4. 默认封禁时长为 180 天
5. 已在有效小黑屋中的股票不会重复新增
6. 重复执行同一区间时结果保持稳定
7. 失败场景显式暴露，不做静默 fallback
